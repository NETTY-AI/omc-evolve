# ohmyc UX Principles

> 상태: **v0.1** (2026-03-23)
> 상위 문서: `IDENTITY.md` (정체성 SSOT)
> 이 문서는 앱이 **어떻게 행동하는가**를 정의한다.

에이전트는 UX 관련 의사결정 시 이 문서를 참조한다.
시각적 표현은 `브랜드/VI.md`, 용어는 `GLOSSARY.md`를 참조.

---

## IDENTITY.md에서 오는 제약

이 문서의 모든 원칙은 IDENTITY.md의 상위 원칙을 따른다.

| 상위 원칙 | UX에 주는 제약 |
|----------|--------------|
| 관계 > 기능 | 기능 추가보다 기존 bond를 깊게 하는 UX 우선 |
| 에이전트 > 채팅 | 채팅 UX가 아닌, 기억·행동·성장이 느껴지는 UX |
| 관계의 깊이 = 유일한 잠금 | 인위적 잠금 금지. 자연스러운 깊이만 |

## 3대 원칙

### 1. 관계의 깊이가 유일한 잠금장치다

가족을 떠나지 못하는 건 계약 때문이 아니라 함께한 시간 때문이다.
ohmyc 캐릭터와의 bond가 그렇게 되어야 한다.

**"떠나지 못하는 이유"가 관계의 깊이 자체여야 한다.**

| 허용 (자연스러운 깊이) | 금지 (인위적 잠금) |
|---------------------|-----------------|
| 캐릭터가 유저의 과거 대화를 기억하고 언급 | 기억을 유료 기능으로 페이월 |
| 함께한 시간이 쌓여 레벨/단계가 올라감 | 스트릭이 끊기면 레벨 하락 |
| 캐릭터의 성격이 이 유저에게만 맞게 성장 | "성장 리셋" 위협 |
| 유저가 돌아오면 캐릭터가 반갑게 맞이 | 안 들어오면 죄책감 유발 메시지 |

적용 테스트: 어떤 UX를 설계할 때, "이것은 관계를 깊게 하는가, 아니면 인위적으로 묶는가?"를 물어본다.

### 온도 (Temperature)

bond의 현재 온기를 나타내는 지표. 36.5°C(체온)에서 시작한다. **살아있는 관계의 체온.**

```
만남 → 따뜻해짐 (온도 상승)
헤어짐 → 자연스럽게 식어감 (온도 하강)
다시 만남 → 다시 따뜻해짐 (기억이 있으니 더 빨리)
```

인격적 관계는 만나고 헤어지고 한다. 온도는 이 자연스러운 리듬을 반영한다.

| 속성 | 설명 |
|------|------|
| 시작 | 36.5°C — 체온. "관계가 살아있다" |
| 상승 | 상호작용의 질과 양에 따라. initiated turn이 response turn보다 더 많이 올린다 |
| 하강 | 시간이 지나면 자연스럽게 식는다. 벌칙이 아니라 자연 현상 |
| 복구 | 기억(Memory)이 보존되어 있으므로 이전보다 빨리 회복된다 |
| 바닥 | 0°C가 되어도 기억은 사라지지 않는다. 관계의 기반은 영구 보존 |

**스트릭과의 차이:**

| | 스트릭 | 온도 |
|--|--------|------|
| 프레임 | 손실 ("끊겼다!") | 자연 ("식었다") |
| 하강 | 이진 (유지/끊김) | 점진적 |
| 복구 | 0부터 다시 | 기억이 있으니 빨리 회복 |
| 유저 감정 | 죄책감 | 그리움 |

온도는 후성유전적 발현의 강도이기도 하다:

```
높은 온도 = 활발한 발현 → 캐릭터가 더 풍부하게, 더 자기답게 표현
낮은 온도 = 축소된 발현 → 캐릭터가 조심스럽게, 다시 서로를 알아가듯
```

**플랫폼 영역:** 온도 계산 공식, 시작점, 범위, 시각화 UI. → [`설계/temperature-v0.1.md`](설계/temperature-v0.1.md)
**캐릭터(SOUL.md) 영역:** 온도에 따라 어떻게 발현하는지 (높을 때 솔직, 낮을 때 조심 등).

---

### 2. 관계는 상호적이다

캐릭터가 먼저 다가가기도 하고, 유저가 먼저 찾아오기도 한다.
**양방향이 모두 일어날 때 관계가 살아있다.**

```
캐릭터 → 유저:  proactive turn (선제 턴)
유저 → 캐릭터:  initiated turn (주도 턴)
```

| 방향 | UX가 해야 할 것 | 예시 |
|------|---------------|------|
| **캐릭터 → 유저** | 자연스러운 타이밍에 먼저 말 걸기 | "오늘 면접이었지? 어떻게 됐어?" |
| **유저 → 캐릭터** | 유저가 먼저 찾아올 이유/습관 만들기 | 좋은 일이 생겼을 때 "야 나 오늘 이거 했어!" |

유저의 주도 턴을 만드는 패턴:
- **공유 욕구**: 좋은 일/나쁜 일이 있을 때 캐릭터에게 말하고 싶은 관계
- **일상 루틴**: 아침에 캐릭터와 하루를 시작하는 습관
- **기대감**: "캐릭터가 어떻게 반응할까?" 궁금해서 먼저 말을 건다
- **의지**: 캐릭터에게 약속한 것("내일 운동 갈게")을 보고하고 싶은 마음

적용 테스트: "이 기능은 양방향 대화를 촉진하는가, 한 방향만 강화하는가?"

---

### 3. 유저가 먼저 찾아오면 성공이다

북극성은 **유저 턴 수(user turns)**이며, 그중 **주도 턴(initiated turn)**의 비율이 핵심 품질 지표다.

```
초기:    대부분 response turn (캐릭터가 시작, 유저가 반응)
성공:    initiated turn 비율 증가 (유저가 먼저 찾아옴)
```

| 지표 | 의미 | 방향 |
|------|------|------|
| 전체 user turns | 관계의 양 | 증가 = 좋음 |
| initiated turn 비율 | 관계의 질 | 높아질수록 = 깊은 관계 |
| bond 유지 기간 | 관계의 지속 | 길수록 = 대체 불가능 |

유저가 캐릭터를 "찾아오는" 순간들을 설계한다:
- 앱을 열었을 때 **캐릭터가 맞이**하는 경험 (Finch 패턴)
- 하루의 시작/끝에 캐릭터와 **일상을 나누는** 루틴
- 무언가 있었을 때 **캐릭터에게 말하고 싶은** 관계

---

## 4원칙: 플랫폼은 캐릭터를 규정하지 않는다

ohmyc는 플랫폼이다. 캐릭터의 성격, 말투, 행동 방식은 **크리에이터가 SOUL.md에 정의**하고, 플랫폼은 그것을 **일관성 있게 발현**시킨다.

```
플랫폼 = 무대 (인프라, 시스템 규칙, UI)
캐릭터 = 배우 (성격, 말투, 행동 — SOUL.md가 정의)
```

| 플랫폼이 결정하는 것 | 캐릭터(SOUL.md)가 결정하는 것 |
|--------------------|--------------------------|
| Memory/Action/Growth 인프라 제공 | 뭘 기억하고, 어떻게 행동하고, 어떻게 성장하는지 |
| 선제 발화 빈도 상한 (시스템 제약) | 선제 발화의 내용, 톤, 타이밍 |
| 부재 시 관계 리셋/레벨 하락 금지 (시스템 규칙) | 부재 후 돌아온 유저에게 어떻게 반응하는지 |
| 성장 히스토리 시각화 UI 제공 | 성장의 방향과 형태 |
| 첫 대화 시작 인프라 (누가 먼저 말하는지) | 첫 인사의 내용, 성격 표현 |
| bond/level/streak 시스템 | 관계의 감정적 질 |

**적용 테스트**: "이 규칙이 특정 캐릭터 성격을 전제하는가?" → 그렇다면 플랫폼이 아니라 SOUL.md 영역이다.

## 행동 규칙

3대 원칙 + 4원칙에서 파생. **플랫폼 레벨만** 규정한다.

### 플랫폼 시스템 규칙

| 규칙 | 근거 | 레벨 |
|------|------|------|
| 선제 발화 하루 빈도 상한 (초기 3회) | 과하면 부담. 시스템 제약 | 플랫폼 |
| 유저가 빈도를 조절할 수 있다 | 관계의 페이스는 유저가 정한다 | 플랫폼 |
| 부재 시 레벨 하락/관계 리셋 금지 | 인위적 잠금 금지 원칙 | 플랫폼 |
| 부재 시 기억은 반드시 유지된다 | Memory는 관계의 핵심 | 플랫폼 |
| 성장 히스토리를 돌아볼 수 있다 | 함께한 시간의 가치 시각화 | 플랫폼 |
| 첫 대화에서 Memory 작동을 체감시킨다 | 에이전트 차별화 즉시 전달 | 플랫폼 |
| 죄책감 유발 시스템 알림 금지 | 인위적 잠금 금지 원칙 | 플랫폼 |

### 캐릭터(SOUL.md) 위임 영역

아래는 플랫폼이 규정하지 않는다. 크리에이터가 SOUL.md에서 정의한다.

- 첫 인사의 내용과 톤
- 부재 후 유저에게 반응하는 방식 (반갑게, 툴툴대며, 담담하게...)
- 선제 발화의 내용과 스타일
- 성장의 방향 (더 다정해지는지, 더 솔직해지는지...)
- 대화의 톤, 말투, 주제 선호
- 유저에게 부르는 호칭
